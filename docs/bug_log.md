## Traps

A **trap** is a failure mode that will fool a future agent — most often the "**green everywhere, broken where nobody looks**" kind, where the type-checker, linter, tests, and build all pass but the thing is still wrong. The traps log is where these are recorded once, canonically, so a close-out entry can *point* to a trap instead of restating it.

**How to write a trap:** give it an id (`TRAP-NNN`) and a one-line description of the deceptive failure, then two things — **why every guard misses it** (walk typecheck → lint → tests → build and name where it actually surfaces, usually only when the real artifact runs) and **the tell** (the concrete check that *does* catch it, since the automated guards won't). Keep the traps themselves here in this doc.

### TRAP-001 — `npm ci` green locally, red in CI: lockfile generated under a different npm major

First hit: T-002's first PR (2026-07-22). `package-lock.json` was generated by npm 11 (Node 26, the local machine); CI ran Node 22's npm 10, which computed a different ideal tree (`vitest@4` needs `esbuild@^0.28`; the lock only carried Vite 5's `esbuild@0.21.5`) and rejected the lock as out of sync (`EUSAGE`).

**Why every guard misses it:** typecheck, lint, tests, build, and even a local `npm ci` all pass — the lock is perfectly valid *for the npm that wrote it*. The failure exists only in the version gap and surfaces only on the CI runner.

**The tell:** the npm major that generates/verifies the lock must be the npm major CI runs. On any `EUSAGE`/lock-sync CI failure, first compare `node --version`/`npm --version` locally vs. the CI pin before debugging dependencies. **And note the fix's own trap:** plain `npm install` can be a no-op — the npm that wrote the bad dedupe considers it complete. Delete `package-lock.json` *and* `node_modules` and re-resolve from scratch (observed on the first fix attempt: npm 11 had deduped vitest 4's nested `vite@8` `esbuild ^0.27||^0.28` requirement onto Vite 5's `esbuild@0.21.5` and saw nothing wrong with it). Standing prevention here: CI pins the human lead's actual toolchain (`docs/devops_pipeline.md`), so local and CI share an npm major.
